From Vanilla to NavLogic

Understanding why satisfies/3 is a meta-interpreter — not just another predicate

Part 1 — What You Already Know

The Vanilla Meta-Interpreter

You've seen this in class. The vanilla meta-interpreter is a Prolog program that interprets Prolog itself. It takes a goal — not as something to execute, but as a data term to inspect — and decides what to do by pattern-matching on its structure:

solve(true). solve((A, B)) :- solve(A), solve(B). % conjunction: solve both solve((A ; B)) :- solve(A) ; solve(B). % disjunction: solve either solve(G) :- clause(G, Body), solve(Body). % atom: look up clause, recurse

The magic word is level shift. In normal Prolog, when you write parent(X, Y), ancestor(Y, Z), the comma is Prolog's own conjunction — the engine handles it. But in the meta-interpreter, (A, B) is a functor being inspected as data. The interpreter sees the comma, decides it means "solve both sides," and recurses. The logic is the same, but the level of control has shifted: from the engine to your code.

The Key Idea

A meta-interpreter treats logical formulas as data and evaluates them by structural recursion. It reads the shape of the formula (is it a conjunction? a disjunction? an atom?) and acts accordingly.

Part 2 — The Parallel

NavLogic's satisfies/3 Does the Same Thing — for a Different Language

The vanilla interpreter evaluates Prolog goals. NavLogic's satisfies/3 evaluates its own goal grammar — a tree of operators like and or not any_tag none_tag prefer_tag that arrives from the LLM as a data term. Same technique, different object language.

Vanilla Meta-Interpreter
solve(true). solve((A, B)) :- solve(A), solve(B). solve(\+ G) :- \+ solve(G). solve(G) :- clause(G, Body), solve(Body).
NavLogic's satisfies/3
% (prefer_tag always true — like "true") satisfies(_, prefer_tag(_), _). satisfies(Poi, and(Gs), T) :- forall(member(G,Gs), satisfies(Poi, G, T)). satisfies(Poi, not(G), T) :- \+ satisfies(Poi, G, T). satisfies(Poi, any_tag(Ts), T) :- member(Tag, Ts), bind_tag(Tag, B), ... tag_active_at(Poi, B, T).

The Eureka Moment

In the vanilla interpreter, (A, B) is data shaped like a comma — the interpreter sees the comma and decides to recurse. In NavLogic, and([G1, G2, G3]) is data shaped like "and" — the interpreter sees the and functor and decides to recurse. Same pattern. Same technique. Different language being interpreted.

The vanilla interpreter interprets Prolog. satisfies/3 interprets the LLM's constraint language. Both work by structural recursion over data terms. That's what makes them both meta-interpreters.

Part 3 — Watch It Walk the Tree

Evaluating a Real Goal, Node by Node

Let's trace a concrete goal through satisfies/3. The user asked for "a temple, but nothing weather-exposed." The LLM produced this AST:

satisfies(wat_mangkon, and([any_tag([temple]), none_tag([weather_exposed])]), t(sat, 20, 0)).
Goal AST — The Data Tree That satisfies/3 Walks
Each node is a functor the meta-interpreter pattern-matches on

Here's exactly what the meta-interpreter does at each numbered node. Notice: at every step, it pattern-matches on the functor of the data term, just like the vanilla interpreter pattern-matches on (A, B) vs. (A ; B) vs. atom.

Pattern-match: and([...]) functor
The second argument is and([any_tag([temple]), none_tag([weather_exposed])]). The interpreter matches the clause satisfies(Poi, and(Gs), T) and enters forall over the list Gs — meaning every child must succeed.
satisfies(wat_mangkon, and([G1, G2]), t(sat,20,0)) :- forall(member(G, [G1, G2]), satisfies(wat_mangkon, G, t(sat,20,0))).
Recurse into child 1: any_tag([temple])
forall picks G = any_tag([temple]) and calls satisfies/3 again. This time, the interpreter matches satisfies(Poi, any_tag(Ts), T) — a leaf clause that grounds against the knowledge base.
Leaf: ground against the KB
At the leaf, the interpreter calls bind_tag(temple, temple) — Clause 1 of bind_tag/2 fires via synonym('temple', temple), cut commits. Then tag_active_at(wat_mangkon, temple, t(sat,20,0)) checks: does wat_mangkon have temple? Yes — tagged(wat_mangkon, [temple, indoor, ...]) and is_a(temple, temple) by reflexivity.
% This is the "clause(G, Body)" moment of the vanilla interpreter. % Instead of looking up a Prolog clause, we look up a tag in the KB. bind_tag(temple, temple), % synonym binding tag_active_at(wat_mangkon, temple, T). % KB lookup — succeeds ✓
Recurse into child 2: none_tag([weather_exposed])
forall picks the second child. The interpreter now matches satisfies(Poi, none_tag(Ts), T) — the negation leaf. This clause asserts that no tag in the list may be present.
Leaf: negated KB lookup
bind_tag(weather_exposed, weather_exposed) — Clause 2 of bind_tag/2, it's a known_tag. Then \+ tag_active_at(wat_mangkon, weather_exposed, T): does wat_mangkon have any tag that is_a weather_exposed? Its tags are [temple, indoor, religious_site, low_noise, mid_budget]. None of these have an is_a path to weather_exposed. Negation-as-failure succeeds.
bind_tag(weather_exposed, weather_exposed), % known_tag binding \+ tag_active_at(wat_mangkon, weather_exposed, T). % indoor ≠ weather_exposed, temple ≠ weather_exposed ... all fail % \+ failure = success ✓ (none_tag semantics: absence = pass)
Result: both children succeeded → and succeeds
forall verified that every member of Gs satisfies. The root and node returns true. Wat Mangkon Kamalawat is a valid candidate for this goal at this time.
Part 4 — Why This Matters

What the Meta-Interpreter Gives Us That call/1 Never Could

A student might ask: "Why not just use Prolog's built-in call/1 to execute the goal directly?" Three reasons — and they're all consequences of the level shift:

1. We Thread Extra Context

Every recursive call carries Poi and Time. The vanilla call/1 has no way to inject these parameters into every level of a conjunction. The meta-interpreter threads them naturally because it controls the recursion.

2. We Can Treat the Same Node Differently in Different Contexts

prefer_tag is vacuously true during satisfies/3 (never blocks a candidate), but preference_score/4 walks the same AST and counts actual matches. One data structure, two interpretations. With call/1, a goal either succeeds or fails — you can't give it a different semantics in a different context.

3. We Can Structurally Transform the Goal

relax/4 uses select/3 to remove conjuncts from the and list and re-evaluate. collect_tags/2 walks the tree to extract all tag literals. sub_goal/2 enumerates all sub-formulas for the audit. None of these are possible if the goal is opaque bytecode inside call/1 — they require the goal to be inspectable data, which is exactly what the meta-interpreter provides.

The One Sentence to Remember

A meta-interpreter is what you get when you stop executing logic and start reading it. The vanilla interpreter reads Prolog's own syntax. NavLogic's satisfies/3 reads the LLM's constraint language. Both work by looking at the shape of a term and recursing — the only difference is what language they're interpreting.
